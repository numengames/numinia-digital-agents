# MIS-055 — Protocolo de Approval Brief

> **Resumen:** Implementar un estándar de comunicación para que los agentes digitales soliciten aprobación a agentes biológicos de forma clara, estructurada y consciente.
> **Epistémico:** Qué principios gobiernan la relación de aprobación humano–máquina en la CAO.
> **Pragmático:** Protocolo activo en todos los agentes de la CAO.
> **Audiencia:** Agentes · Oráculos

---

## Historia

Como Oráculo, quiero que cuando un agente digital me pida aprobación, entienda exactamente qué va a hacer, por qué, y cuánta atención debo prestarle — para poder decidir de forma informada y consciente sin fricción innecesaria.

## Criterios de aceptación

- [x] P-008 documentado en `protocols/P-008-approval-brief-v1.md`
- [x] Escala 1–10 adoptada del canon (APPROVAL-REQUEST-template.md)
- [x] Campo `misión` obligatorio en cada approval
- [x] Tipos de aprobación: Ejecución + Diseño UX/UI
- [x] Principio de responsabilidad explícito y no negociable
- [x] OPERATOR.md del agente actualizado: git pull como paso obligatorio de startup
- [ ] QA del Oracle: revisión de las primeras 10 aprobaciones bajo nuevo formato
- [ ] Todos los agentes activos aplican el mismo protocolo (verificar en Dark Council)

## Lección aprendida (inscrita en la misión)

Al implementar esta misión se creó primero un protocolo local (C1–C5) sin consultar el repo. El repo ya tenía `APPROVAL-REQUEST-template.md` con un sistema más completo. Hubo que reescribir para alinearlo al canon.

**Regla operativa derivada:**
1. `git pull` al inicio de cada sesión
2. Revisar si el repo ya tiene lo que se necesita
3. Si existe: derivar. Si no existe: crear y subir al repo.
