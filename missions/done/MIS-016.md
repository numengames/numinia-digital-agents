## 📖 Historia

Como operador, quiero todos los servicios accesibles via HTTPS con subdominios propios, para que nada sea accesible por IP:puerto.

## 🧠 Valor Epistémico

- **Hipótesis:** Nginx + Certbot es el stack estándar para reverse proxy + SSL en Linux
- **Resultado real:** Caddy es superior para este caso — SSL automático sin Certbot, configuración mínima
- **Aprendizaje:** Para stacks modernos sin requisitos especiales de Nginx, Caddy elimina fricción operativa

## ⚡ Valor Pragmático

Infraestructura profesional. Los links son compartibles. Nada accesible por IP:puerto.

## ✅ Criterios de aceptación (planificados)

- [x] Reverse proxy instalado
- [x] SSL via Let's Encrypt para: analytics.pablofm.com, cal.pablofm.com
- [x] Puertos 3001 y 3002 solo accesibles via Tailscale (no expuestos públicamente)
- [x] Renovación SSL automática configurada

---

## 🏗️ Ejecución Real
*(Los planos reales vs los planos ideales)*

- **Tecnología planificada:** Nginx + Certbot (manual)
- **Tecnología ejecutada:** Caddy (reverse proxy moderno)
- **Por qué divergió:** Caddy gestiona Let's Encrypt de forma completamente automática — sin Certbot, sin cronjobs, sin configuración manual de renovación. Nginx requería Certbot + configuración adicional para el mismo resultado.
- **Resultado adicional no planificado:** Firewall configurado con Tailscale — los puertos 3001/3002 son accesibles solo por Pablo vía Tailscale, no expuestos públicamente. El público accede via Caddy (puerto 443). Arquitectura más segura que la planificada.
- **Aprendizaje clave:** Cuando el stack no tiene requisitos específicos de Nginx (módulos custom, configuraciones avanzadas), Caddy es la opción superior para operadores que priorizan mantenimiento mínimo.
